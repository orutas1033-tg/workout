# 筋トレメモ

ワークアウトと種目別の記録を、**端末内だけ**で管理するメモアプリ。
単一の HTML ファイルで動き、外部サーバーへの通信は一切行わない。

- 要件の詳細は `requirements.md` を参照。
- 実装は段階的に進めている。**現在はフェーズ1（データ層 + 種目マスタ + バックアップ）まで完了。**

## 動かし方

### A. Mac でとりあえず確認する

`index.html` をダブルクリックしてブラウザで開くだけで、種目登録もバックアップも動く。

ただし `file://` で開くと Service Worker が登録されないため、オフラインキャッシュだけは効かない
（アプリのロジック自体には影響しない）。
ローカルで PWA としての挙動まで確認したい場合は、簡易サーバーを立てる：

```bash
# ファイルのあるディレクトリで実行
cd /path/to/kintore-memo

# Python の標準ライブラリだけでローカルサーバーを起動（8000番ポート）
python3 -m http.server 8000
```

- `python3 -m http.server 8000`
  - `-m http.server` … Python 標準の簡易 HTTP サーバーモジュールを実行する
  - `8000` … 待ち受けポート番号
  - 起動後、ブラウザで `http://localhost:8000/` を開く
  - 停止は `Ctrl + C`

### B. iPhone で使う（本番）

GitHub Pages に置いて、Safari からホーム画面に追加する。

1. GitHub で**公開リポジトリ**を作る（例: `kintore-memo`）
   - 公開されるのは**アプリのコードだけ**で、トレーニング記録は含まれない。
     記録はブラウザの localStorage にしか存在せず、リポジトリには一切上がらない。
2. `index.html` / `manifest.json` / `sw.js` をアップロードする
3. リポジトリの **Settings → Pages** で
   - Source: `Deploy from a branch`
   - Branch: `main` / `/ (root)`
   を選んで Save
4. 数分待つと `https://<ユーザー名>.github.io/kintore-memo/` で公開される
5. **iPhone の Safari** でその URL を開く
6. 共有ボタン → **「ホーム画面に追加」**

> ⚠️ 手順6は必ず **Safari** で行うこと。Chrome など他ブラウザからはホーム画面追加が正しく機能しない。

## ⚠️ データの保存について（必ず読むこと）

### 記録は「端末のブラウザの中」にしかない

- 保存先は **localStorage**（キー: `kintore-memo`）。クラウドにもサーバーにも保存されない。
- **localStorage は永続保証がない。** 次の場合に記録は消える：
  - Safari の「履歴と Web サイトデータを消去」を実行した
  - iOS の ITP（トラッキング防止機能）がストレージを削除した
    ※ ホーム画面追加した PWA は対象外とされているが、**Apple が永続性を保証しているわけではない**
  - 端末を初期化した / 機種変更した
- **保存先の URL（オリジン）が変わると記録は引き継がれない。**
  GitHub Pages のリポジトリ名を変えたり、別の場所へ引っ越すと、記録は見えなくなる。

### したがって

**設定タブから定期的に「JSONを書き出す」こと。** これが唯一の復元手段。
最後の書き出しから14日経つと、アプリが警告バーを出す。

- 書き出した JSON は iPhone の「ファイル」アプリ → iCloud Drive などに保存する
- 復元は 設定タブ → 「JSONから復元する」
- **CSV は復元には使えない**（種目マスタが含まれないため）。解析用。

## CSV の形式（解析用）

1行1セットのフラット形式。`pandas.read_csv()` でそのまま読める。

| 列名 | 内容 |
|---|---|
| `date` | ワークアウト実施日（YYYY-MM-DD） |
| `workout_duration_min` | ワークアウト全体の所要時間（分）。持久系種目の時間とは別物 |
| `exercise_name` | 種目名（その時点の表示名） |
| `kind` | `strength` / `endurance` |
| `set_no` | セット番号（1始まり） |
| `weight_kg` | 重量。片方ずつの種目では**片側の**重量 |
| `reps` | 回数。片方ずつの種目では**片側の**回数 |
| `is_unilateral` | 片方ずつ行ったか（1 / 0） |
| `rpe` | 主観的なキツさ（1–10、0.5刻み）。**任意入力なので空欄がありうる** |
| `duration_sec` | 持久系の実施時間 |
| `distance_km` | 持久系の実施距離 |
| `volume` | `weight_kg × reps × (is_unilateral ? 2 : 1)` ※左右合算 |

BOM 付き UTF-8 で出力しているため、Excel でも文字化けしない。
pandas で読む場合は `encoding="utf-8-sig"` を指定する：

```python
import pandas as pd
df = pd.read_csv("kintore-2026-07-13.csv", encoding="utf-8-sig")
```

## 設計上の注意点（コードを読む前に）

- **種目は物理削除できない。** 過去のセット記録が `exerciseId` で種目を参照しているため、
  削除すると履歴が壊れる。使わなくなった種目は「記録画面の選択肢に表示する」のチェックを外す。
- **種目名はリネームしてよい。** 内部 ID で参照しているので、名前を変えても履歴は分断されない。
- **推定1RM は参考値。** Epley 式（`weight × (1 + reps/30)`）は高レップからの外挿で誤差が大きい。
  フェーズ3で表示する際も、その旨を UI に明記する。
- **消費カロリーは意図的に実装していない。** 理由は `requirements.md` の第5章。
